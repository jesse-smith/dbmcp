## Items

### Robustness — `src/analysis/column_stats.py`

- [ ] **WR-01 — bare `except Exception: return None` masks real failures.**
  `_try_describe_extended_stats` (column_stats.py:425-429) treats auth failures, network
  errors, injection-induced syntax errors, and "stats genuinely unavailable" identically,
  then silently falls through to the Tier-2 aggregate path. Hides operational failures.
  **Fix:** narrow the catch to `sqlalchemy.exc.SQLAlchemyError` (and ideally inspect to
  distinguish "DESCRIBE EXTENDED unsupported" from infra errors that should propagate).
  *Of the 7, this is the one most worth promoting to urgent if it ever masks a real incident.*

- [ ] **WR-02 — `row[1]`/`row[2]` dereferenced after only guarding `row` truthiness.**
  `get_basic_stats`/`get_numeric_stats`/etc. (column_stats.py:265-269, 293-308, 375-380) use
  `row[0] if row else 0` / `row[1] if row else 0`; the `else 0` only protects against `None`,
  not against a short row. A differently-shaped aggregate result raises `IndexError` instead
  of falling back. Cross-catalog transpile work makes shape assumptions riskier.
  **Fix:** either drop the misleading `if row else` guards (trust the aggregate contract) or
  guard width explicitly: `row[1] if row and len(row) > 1 else 0`.

### Clarity / dead code — `src/analysis/column_stats.py`

- [ ] **WR-05 (Phase 15.1) — Databricks DESCRIBE EXTENDED "fast path" is dead code cross-catalog.**
  `get_columns_info` (column_stats.py:600-623) probes only `columns_to_analyze[0]` to decide
  `use_fast_path`. On the cross-catalog branch `get_column_data_type` returns a *string*
  (column_stats.py:194-198), so `isinstance(type_info, sa_types.TypeEngine)` is always False
  and the fast path never fires — every cross-catalog column silently takes the Tier-2
  aggregate path. Functionally fine, but contradicts the module/tool docstrings that advertise
  a Databricks fast path.
  **Fix:** either gate the probe on `not self._is_cross_catalog_databricks` and document the
  fast path as default-catalog-only, or make the cross-catalog reflector return a `TypeEngine`
  so the fast path can engage. Also drop the single-column probe in favor of the already-
  per-column fall-through (or document why column 0 is representative).

### Dedup / consistency — `src/mcp_server/analysis_tools.py` (IN-02/03/04 share one fix)

- [ ] **IN-03 — identical cross-catalog scaffolding duplicated across all three tools.**
  The `cross_catalog = bool(resolved.catalog) and dialect is not None and dialect.name ==
  "databricks"` computation plus the `if cross_catalog: MetadataService.table_exists(...) else:
  inspector.get_table_names/get_view_names` existence block is copy-pasted at
  analysis_tools.py:112-147, 262-295, 429-462 (the same predicate is also independently
  recomputed inside `PKDiscovery`/`FKCandidateSearch`/`ColumnStatsCollector`). Rule of Three
  crossed.
  **Fix:** extract a shared `resolve_and_check_table_exists(...)` helper (and ideally the
  `cross_catalog` predicate) used by all three tools. **This helper naturally subsumes IN-02
  and IN-04 below** — do them together.

- [ ] **IN-02 — lazy `from src.db.metadata import MetadataService` duplicated in 3 tool fns.**
  analysis_tools.py:126-128, 275-277, 442-444. Either hoist to module top-level (if no import
  cycle) or add a one-line comment naming the cycle being broken. Folds into the IN-03 helper.

- [ ] **IN-04 — inconsistent "table not found" message between cross-catalog and default paths.**
  Cross-catalog emits `Table 'schema.table' not found`; default emits
  `Table 'table' not found in schema 'schema'` (analysis_tools.py:135 vs 146, 284/294, 451/461).
  Same failure, two formats — downstream error_message parsers see drift. Use one template for
  both branches. Folds into the IN-03 helper.

### Consistency — `src/analysis/_sql.py`

- [ ] **IN-01 — `list_tables` row-shape fallback contradicts its documented contract.**
  `_sql.py:119` (now ~140 after the WR-03 fix): `return [row[1] if len(row) > 1 else row[0]
  for row in rows]`. The docstring says SHOW TABLES returns `(database, tableName, isTemporary)`
  with `row[1]` the table name; the `len(row) > 1` fallback to `row[0]` would silently return a
  database name as a table name for a shape the docstring says never occurs.
  **Fix:** assert the expected shape (fail fast) or document why the fallback exists.
