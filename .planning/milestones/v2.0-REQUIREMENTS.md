# Requirements Archive: v2.0 Multi-Dialect Support

**Milestone:** v2.0
**Shipped:** 2026-05-06
**Defined:** 2026-04-13

All 26 v2.0 requirements satisfied. This archive records the final status per-requirement as verified in phase `*-VERIFICATION.md` tables and cross-referenced in `v2.0-MILESTONE-AUDIT.md` (passed 2026-05-06).

## Core Value

LLM agents can explore and query databases safely, with validated read-only access, dialect-aware metadata, and clear error reporting.

## Requirements (26/26 satisfied)

### Dialect Infrastructure (5/5)

- [x] **DIAL-01**: Server supports DialectStrategy protocol with name, sqlglot_dialect, create_engine, fast_row_counts, quote_identifier, and capability flags — Phase 8
- [x] **DIAL-02**: MssqlDialect extracts all existing SQL Server-specific code (ODBC strings, Azure AD, DMV queries) behind the protocol — Phase 8
- [x] **DIAL-03**: DatabricksDialect builds databricks:// engines with token auth, catalog/schema awareness, and Databricks-optimized paths — Phase 11
- [x] **DIAL-04**: GenericDialect accepts any SQLAlchemy URL and uses Inspector-only metadata with COUNT(*) fallback for row counts — Phase 10
- [x] **DIAL-05**: Dialect registry maps dialect names to strategy implementations (GenericDialect as fallback, deferred from Phase 8 to Phase 10) — Phase 8/10

### Connection & Config (5/5)

- [x] **CONF-01**: TOML config supports `dialect` discriminator field — Phase 9
  - **Adjusted during execution (D-02):** `dialect` is **required**, not defaulted to "mssql". The original "backward compatible default" was abandoned — explicit failure is clearer for multi-dialect config. ROADMAP success-criterion and this row updated 2026-05-06.
- [x] **CONF-02**: Typed config models validate dialect-specific fields (MssqlConnectionConfig, DatabricksConnectionConfig, GenericConnectionConfig) — Phase 9
- [x] **CONF-03**: connect_database tool accepts connection_name or sqlalchemy_url (clean break — old SQL Server-specific params removed) — Phase 10
- [x] **CONF-04**: pyodbc and azure-identity move to `mssql` optional extra; databricks packages to `databricks` extra — Phase 10
- [x] **CONF-05**: Dialect-specific dependencies use lazy imports with clear error messages when missing — Phase 10

### Query Validation (3/3)

- [x] **VALID-01**: validate_query accepts dialect parameter and passes it to sqlglot.parse() — Phase 9 (integration closed in Phase 13.1)
- [x] **VALID-02**: Safe procedure list is dialect-aware (MSSQL sp_ list; empty for Databricks/generic) — Phase 9 (integration closed in Phase 13.1)
- [x] **VALID-03**: Denylist validation (INSERT/UPDATE/DELETE/CREATE/DROP) works unchanged across all sqlglot dialects — Phase 9

### Metadata (5/5)

- [x] **META-01**: list_schemas, list_tables, get_table_schema work for all three dialects via Inspector with MSSQL optimized overrides preserved — Phase 11 (connect_database integration closed in Phase 13.1)
- [x] **META-02**: Databricks three-level namespace (catalog.schema.table) scoped per connection with catalog in data model — Phase 11
- [x] **META-03**: Databricks table properties surfaced (owner, storage format, managed/external, creation time) via DESCRIBE EXTENDED — Phase 11
- [x] **META-04**: get_table_schema omits index section for dialects where supports_indexes is false (Databricks) — Phase 11
- [x] **META-05**: Dialect-appropriate identifier quoting in all generated SQL (brackets/backticks/double-quotes) — Phase 8 (get_sample_data + ConnectionManager integration closed in Phase 13.1)

### Analysis Tools (5/5)

- [x] **ANLYS-01**: get_column_info works across all dialects using standard SQL aggregates (Tier 2) with sqlglot transpilation — Phase 12
- [x] **ANLYS-02**: Databricks get_column_info reads precomputed stats from DESCRIBE EXTENDED when available (Tier 3) — Phase 12
- [x] **ANLYS-03**: find_pk_candidates works across all dialects using uniqueness/null checks (Tier 2), with informational-constraint awareness for Databricks — Phase 12
- [x] **ANLYS-04**: find_fk_candidates works across all dialects using Inspector-based index checks and value overlap via INTERSECT (Tier 2) — Phase 12
- [x] **ANLYS-05**: Databricks partition metadata surfaced in table schema responses — Phase 12

### Testing (3/3)

- [x] **TEST-01**: All existing MSSQL tests pass unchanged after dialect extraction (zero behavior change) — Phase 8
- [x] **TEST-02**: Dialect-parameterized test fixtures for generic and Databricks paths (mock-based, no live connection required) — Phase 13
- [x] **TEST-03**: 85%+ test coverage maintained across all modules — Phase 13
  - **Adjusted during execution:** original wording was "70%+"; coverage floor was ratcheted to 85% in Phase 13-04 (baseline 90.64%). Wording updated in REQUIREMENTS.md during Phase 13.1.

## Out of Scope (verified)

All v2.0 out-of-scope items held:

| Feature | Reason | Verdict |
|---------|--------|---------|
| Write operations for any dialect | Violates core read-only security model | ✓ Held |
| Auto-triggering ANALYZE TABLE | Runs compute on user's cluster, costs money | ✓ Held |
| Full cross-catalog queries | Massive complexity; scope to one catalog per connection | ✓ Held |
| Databricks-specific SQL in generic tools | Leaks dialect knowledge | ✓ Held (kept in DialectStrategy) |
| Index metadata for Databricks | Databricks has no traditional indexes | ✓ Held (supports_indexes gating) |
| Enforced constraint semantics for Databricks PK/FK | Misleading — constraints are informational only | ✓ Held |
| Pydantic migration for data models | Current dataclasses work fine | ✓ Held |
| Query result caching / audit logging | Future milestone | ✓ Held |
