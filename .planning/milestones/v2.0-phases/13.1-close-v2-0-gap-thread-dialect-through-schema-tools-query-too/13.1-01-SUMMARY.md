# Phase 13.1 Plan 01: WIRING-01 (connect_database dialect threading) Summary

One-liner: Threaded the resolved dialect into `connect_database`'s one-shot `MetadataService` so `schema_count` is correct for generic SQLAlchemy URLs whose `engine.dialect.name` differs from the "generic" registry key.

## What changed

**Source (`src/mcp_server/schema_tools.py`, line 170, inside `_sync_connect`):**

```diff
-        metadata_svc = MetadataService(engine)
+        metadata_svc = MetadataService(engine, dialect=dialect)
```

`dialect` is already bound in both branches of `_sync_connect`:
- TOML named-connection branch (line 153): `dialect = dialect_cls()`
- URL branch (line 162): `dialect = resolve_dialect_from_url(sqlalchemy_url)`

So the fix is one line — no other structural changes.

**Tests (`tests/unit/test_connect_tool.py`, appended class `TestConnectDatabaseThreadsDialectIntoMetadataService`):**

1. `test_url_path_threads_dialect_kwarg_into_metadata_service` — construction assert via `MockMS.assert_called_once_with(mock_engine, dialect=fake_dialect)`.
2. `test_schema_count_correct_for_generic_url_with_mismatched_engine_dialect` — behavior assert that `data["schema_count"] == 3` after decoding the TOON response, with `mock_engine.dialect.name = "postgresql"` to pin the previously-degrading scenario.

Both tests patch `src.mcp_server.schema_tools.MetadataService`, `resolve_dialect_from_url`, `get_connection_manager`, `get_config`, and `Path` — mirroring existing test patterns.
