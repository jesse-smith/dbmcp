# Milestone v2.0: Multi-Dialect Support

**Status:** ✅ SHIPPED 2026-05-06
**Phases:** 8-13 (+ 13.1 inserted)
**Total Plans:** 20

## Overview

Extended dbmcp from SQL Server-only to support Databricks and arbitrary SQLAlchemy databases via a dialect strategy pattern. All MCP tools now dispatch dialect-specific behavior through `DialectStrategy`, with zero-behavior-change for existing MSSQL users, optimized Databricks paths (catalog, DESCRIBE EXTENDED, Tier-3 precomputed stats), and a GenericDialect fallback covering any SQLAlchemy URL.

## Milestone Summary

**Decimal Phases:**
- Phase 13.1: Close v2.0 Gap — inserted after Phase 13 to resolve wiring gaps and tech debt surfaced by the 2026-05-05 milestone audit before closing v2.0.

**Key Decisions:**
- **Dialect required, no default** (D-02, Phase 9): TOML `dialect` field is required — no silent "mssql" fallback. Clearer failure mode for multi-dialect config.
- **Write base SQL in TSQL + transpile via sqlglot** (Phase 12): Single canonical query form, transpiled per-dialect — avoids N-way hand-written queries.
- **Probe-first-column heuristic for Databricks DESCRIBE EXTENDED fast path** (Phase 12): Tier-3 precomputed stats read from DESCRIBE EXTENDED when available; falls back to Tier-2 standard SQL otherwise.
- **Inspector-first with MSSQL INFORMATION_SCHEMA fallback** (Phase 12): PK/FK constraint and table discovery uses SQLAlchemy Inspector as the canonical path, MSSQL-optimized fallbacks preserved.
- **`supports_indexes` capability gating** (Phase 11): `get_table_schema` omits index section for Databricks (no traditional indexes); `target_has_index=None` for FK candidate analysis.
- **Coverage floor ratchet 70 → 85** (Phase 13): Single global knob, ~5pt headroom over 90.64% baseline.
- **`build_sample_query` delegated to DialectStrategy** (quick 260506-n8s, post-execution): QueryService's sample-query methods delegate to the dialect, emitting `TOP` for MSSQL and `LIMIT` for Databricks/Generic. Moved dialect-specific SQL generation out of QueryService entirely.

**Issues Resolved:**
- WIRING-01: connect_database's schema_count response threaded through registered dialect (was silently tsql-dispatched for generic URLs)
- WIRING-02: get_sample_data threads dialect into MetadataService and QueryService (was silently tsql-validating non-MSSQL connections)
- WIRING-03: ConnectionManager.connect() accepts optional caller-supplied dialect (latent coupling cleared)
- Dialect-blind TSQL in sample-query SQL generation (Databricks live test failure; resolved via DialectStrategy.build_sample_query)
- Database connection config parse failures now surface to MCP clients instead of crashing server
- Azure AD pool_recycle + pool_pre_ping kept from v1.1, extended to all dialects
- ROADMAP Phase 9 success-criterion text updated to match D-02 (dialect required)
- REQUIREMENTS TEST-03 wording updated to 85%+ (matches implementation floor)
- SUMMARY frontmatter `requirements-completed` reconciled against VERIFICATION SATISFIED rows across all 6 phases
