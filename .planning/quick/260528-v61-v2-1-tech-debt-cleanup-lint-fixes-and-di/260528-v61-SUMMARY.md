### Plan Expectation Correction (RED behavior)

**[Documentation accuracy] The plan predicted the two `None`-expecting tests (Databricks/generic) would fail RED and MSSQL would stay green. The actual RED was inverted.**

- **Found during:** Task C RED run.
- **Root cause:** The bug is in `fk.get("referred_schema", "dbo")`. SQLAlchemy returns the FK dict with `referred_schema` **present but set to `None`**, so `.get()`'s `"dbo"` default never fires â€” it returns the explicit `None`. Under the old code, *every* dialect produced `target_schema=None`; Databricks/generic were correct only by accident, and MSSQL silently lost its `"dbo"`.
- **Consequence:** At RED, the two `None`-expecting tests passed (old code already returned `None`) and the **MSSQL test failed** (`None != "dbo"`). This is still a valid TDD gate: a real regression test fails before the fix and passes after.
- **Fix:** The planned `or`-based expression is exactly right and resolves all three cases. No change to the intended fix; only the RED narrative differs.
