# Phase 15.1: Cross-catalog metadata threading (CR-02 / DISC-01) - Research

**Researched:** 2026-05-29
**Domain:** SQLAlchemy reflection + Databricks Unity Catalog 3-level namespace; cross-dialect identifier threading
**Confidence:** HIGH

## Summary

The CR-02 defect is now fully diagnosed at the source level. The `databricks-sqlalchemy`
dialect (installed **v2.0.9**, against SQLAlchemy **2.0.49**) **hard-binds every public
Inspector reflection method to `self.catalog`**, a value frozen once at connect time from
the engine URL's `?catalog=` query param. The public `schema=` kwarg accepted by
`get_columns`, `get_pk_constraint`, `get_foreign_keys`, and `get_table_names` is a
schema-only qualifier — it is backtick-quoted as a single identifier and concatenated as
`` `{self.catalog}`.`{schema}`.`{table}` ``. Passing `schema="catalog.schema"` would produce
`` `catalog.schema` `` (one quoted token containing a dot), which is wrong. **Option (a) from
the brief — compound `schema="catalog.schema"` to Inspector methods — is RULED OUT.**
`[VERIFIED: installed databricks/sqlalchemy/base.py v2.0.9 lines 117-118, 124-145, 156-175, 197-309]`

This forces the fix to follow the pattern the codebase ALREADY uses for the working
cross-catalog tools (`list_schemas`, `list_tables`, `get_table_schema` columns, `get_sample_data`):
**bypass the Inspector and emit fully-qualified 3-part raw SQL** (`` `catalog`.`schema`.`table` ``),
quoted via `dialect.quote_identifier`. The three target tools split into two mechanical
sub-problems:

1. **Inspector-backed reflection calls** (`get_columns`, `get_pk_constraint`,
   `get_unique_constraints`, `get_foreign_keys`) inside `PKDiscovery`, `FKCandidateSearch`,
   and `ColumnStatsCollector.column_exists/get_columns_by_pattern`. These CANNOT be made
   cross-catalog through the Inspector. They must be re-routed to catalog-aware raw SQL
   (Databricks `DESCRIBE TABLE` / `information_schema` against the explicit catalog) when a
   non-default catalog is resolved.
2. **Structural aggregate SQL** (`_qualified_table = f"[{schema}].[{table}]"`, transpiled
   TSQL→Databricks via sqlglot). This is the EASY half: sqlglot transpiles 3-part TSQL
   bracket identifiers to 3-part Databricks backticks cleanly (verified: `[cat].[sch].[tbl]`
   → `` `cat`.`sch`.`tbl` ``), so threading catalog into `_qualified_table` is a one-segment
   change.

**Primary recommendation:** Thread `resolved.catalog` into the three analysis classes as a
new optional `catalog` constructor arg. On the structural-SQL path, build a 3-part
`_qualified_table` when catalog is set (sqlglot-safe). On the Inspector-backed path, when
`catalog` is set and dialect is Databricks, branch to catalog-aware raw SQL
(`DESCRIBE TABLE <cat>.<sch>.<tbl>` for columns, `information_schema` for
constraints/uniqueness) instead of calling the catalog-blind Inspector — mirroring the
existing `MetadataService._get_databricks_columns` pattern. **Do NOT use `USE CATALOG`** —
it mutates pooled-connection session state and leaks across requests (see Pitfall 1).

### Pattern 1: Catalog-aware raw SQL bypass (the established working pattern)
**What:** When a non-default catalog is requested on Databricks, build a fully-qualified
3-part name with `dialect.quote_identifier` per segment and execute raw `text()` SQL,
bypassing the catalog-blind Inspector.
**When to use:** Every Inspector-backed read in the three analysis classes, on the
cross-catalog branch only. The default-catalog path stays on the Inspector.
**Example (from the codebase — the canonical reference):**
```python
# Source: src/db/metadata.py:945-996  _get_databricks_columns  [VERIFIED: codebase]
quoted_catalog = self._dialect.quote_identifier(catalog)
quoted_schema = self._dialect.quote_identifier(schema_name)
quoted_table = self._dialect.quote_identifier(table_name)
qualified = f"{quoted_catalog}.{quoted_schema}.{quoted_table}"
with self.engine.connect() as conn:
    result = conn.execute(text(f"DESCRIBE TABLE {qualified}"))
    rows = result.fetchall()
# parse rows[i][0]=col_name, rows[i][1]=data_type; stop at blank/"#" marker
```

### Pattern 2: 3-part structural SQL via sqlglot transpilation
**What:** Make `_qualified_table` 3-part on the cross-catalog branch; sqlglot transpiles it.
**When to use:** Structural uniqueness/aggregate queries (`_column_is_unique`, `get_basic_stats`,
numeric/datetime/string stats) that already go through `transpile_query`.
**Example (transpile behavior VERIFIED in this session):**
```python
# Source: verified via `uv run python -c "import sqlglot; ..."`  [VERIFIED: sqlglot local run]
sqlglot.transpile("SELECT COUNT(*) FROM [cat].[sch].[tbl]", read="tsql", write="databricks")[0]
# → "SELECT COUNT(*) FROM `cat`.`sch`.`tbl`"   (clean 3-part backtick output)

# So in pk_discovery / column_stats:
self._qualified_table = (
    f"[{catalog}].[{schema_name}].[{table_name}]" if catalog
    else f"[{schema_name}].[{table_name}]"
)
```

### Pattern 3: Low-level connector catalog targeting (alternative for columns only)
**What:** The DBSQL cursor's `.columns(catalog_name=..., schema_name=..., table_name=...)`
accepts an explicit catalog. The dialect's own `get_columns` hard-codes `catalog_name=self.catalog`,
but the cursor primitive itself is catalog-flexible.
**When to use:** Only if a structured column metadata (typed) result is needed without parsing
`DESCRIBE TABLE` text. For this phase, the existing `DESCRIBE TABLE` parser (Pattern 1) is
already battle-tested and preferred for consistency.
**Source:** `databricks/sqlalchemy/base.py:124-134` shows `cur.columns(catalog_name=self.catalog, ...)`. `[VERIFIED: installed source]`

### Anti-Patterns to Avoid
- **`schema="catalog.schema"` to Inspector methods:** Produces `` `catalog.schema` `` (single
  quoted token). Silently wrong — exactly the class of bug this phase fixes. `[VERIFIED: base.py:175,281,309]`
- **`USE CATALOG x` on a pooled connection:** Mutates session state that survives connection
  return-to-pool; leaks to the next unrelated request. See Pitfall 1.
- **Rebuilding the engine per request:** Defeats QueuePool; violates the one-engine-one-catalog
  IDENT-01 invariant.
- **Threading catalog into the Inspector path silently (the original CR-02 mistake):** accepting
  catalog but binding to default. The fix must either target the requested catalog OR fail loudly.

## Common Pitfalls

### Pitfall 1: `USE CATALOG` leaks across pooled connections
**What goes wrong:** Issuing `USE CATALOG bmtct` to target a non-default catalog mutates the
session's current catalog. When the connection returns to the QueuePool and is checked out by
the next request (possibly a different tool, different catalog), it inherits the stale catalog —
silent cross-request mis-targeting, worse than the bug being fixed.
**Why it happens:** `ConnectionManager` uses SQLAlchemy `QueuePool` (pool_size=5,
max_overflow=10, `pool_pre_ping=True`). `[VERIFIED: src/db/connection.py:127,206-216]` The
default `reset_on_return='rollback'` only rolls back transactions; `USE CATALOG` is a
session/catalog-scope command, not transactional, so it survives the reset.
**How to avoid:** Never issue `USE CATALOG`. Use fully-qualified 3-part names in every
statement (the `get_sample_data` / `MetadataService` approach). Statelessness is the whole
reason those tools work cross-catalog today.
**Warning signs:** Any plan task that says "set the catalog on the connection" or "switch
catalog before reflecting." Reject it.

### Pitfall 2: The Inspector silently returns `[]` for a wrong-catalog table
**What goes wrong:** `inspector.get_columns(table, schema=schema)` against a table that exists
only in a non-default catalog returns an empty list (not an error), because it queries
`self.catalog`. PK/FK discovery then reports "no candidates" — a false negative that looks like
valid empty output.
**Why it happens:** `get_columns` → `cur.columns(catalog_name=self.catalog, ...)`; empty result
triggers a `_describe_table_extended` existence probe against `self.catalog` only.
`[VERIFIED: base.py:124-147]`
**How to avoid:** On the cross-catalog branch, never call the Inspector — go straight to
catalog-aware raw SQL. Add a regression test asserting non-empty results for a table that
exists ONLY in the non-default catalog.
**Warning signs:** Tests that pass with mocked Inspectors returning data regardless of catalog —
they won't catch this. Need a live or catalog-discriminating test (see Validation Architecture).

### Pitfall 3: FK target-table resolution must also be catalog-scoped
**What goes wrong:** `FKCandidateSearch` resolves a SET of target tables
(`get_target_tables` → `_get_target_tables_inspector` uses `inspector.get_table_names(schema=...)`).
If only the source table's reads are threaded but target enumeration stays on the Inspector,
FK candidates are searched in the WRONG catalog's tables.
**Why it happens:** FK search is multi-table; catalog threading must cover both source column
metadata AND the target-table universe.
**How to avoid:** Thread catalog through `get_target_tables` too — use
`SHOW TABLES IN <cat>.<sch>` (the `_list_tables_databricks` pattern) on the cross-catalog branch.
**Warning signs:** FK candidates returned from the default catalog even when the source is
cross-catalog.

### Pitfall 4: `_qualified_table` is bracket-TSQL, not final SQL
**What goes wrong:** Adding a 3-part name as raw backticks into `_qualified_table` would be
double-quoted or mis-transpiled, because that string is fed through `transpile_query(read="tsql")`.
**Why it happens:** The analysis classes write base SQL in TSQL syntax (`[brackets]`) and
transpile to the target dialect. `[VERIFIED: src/analysis/_sql.py + column_stats.py:81]`
**How to avoid:** Build `_qualified_table` as `[catalog].[schema].[table]` (TSQL brackets) and
let sqlglot transpile (verified clean: `` `cat`.`sch`.`tbl` ``). Do NOT pre-quote with backticks.

### Pitfall 5: DESCRIBE EXTENDED fast path in column_stats is 2-part
**What goes wrong:** `ColumnStatsCollector._try_describe_extended_stats` builds
`f"{qi(schema)}.{qi(table)}"` (2-part) and runs `DESCRIBE EXTENDED <2-part> <col>`. On the
cross-catalog branch this hits the default catalog → wrong/empty precomputed stats, then
silently falls through to the aggregate path (which, if also 2-part, is equally wrong).
**Why it happens:** The fast path predates cross-catalog support. `[VERIFIED: column_stats.py _try_describe_extended_stats]`
**How to avoid:** Thread catalog into that fast path too — make it 3-part when catalog is set.

## Open Questions

1. **Should cross-catalog FK target search span catalogs, or stay within the resolved catalog?**
   - What we know: `find_fk_candidates` searches target tables for the referenced side. The
     source column is in `resolved.catalog`. FK relationships in Unity Catalog can cross catalogs.
   - What's unclear: Whether the phase should scope targets to `resolved.catalog` only (simpler,
     matches "target the requested catalog") or allow `target_schema`/`target_tables` to imply a
     different catalog (more general, more surface).
   - Recommendation: Scope target search to `resolved.catalog` for this phase (KISS, matches the
     stated goal of "target the requested catalog"). Cross-catalog FK targets = a deferred idea.

2. **Extract a shared catalog-aware reflection helper, or inline per class?**
   - What we know: `MetadataService._get_databricks_columns` already does catalog-aware column
     reflection; three analysis classes need the same.
   - What's unclear: Whether to extract a `CatalogAwareReflector` helper now or inline.
   - Recommendation: Extract — this is the 4th occurrence (Rule of Three crossed). A small helper
     in `src/db/metadata.py` or `src/analysis/_sql.py` keeps DRY without over-engineering.

### Mock/Unit strategy (the strong assertion without a DB)
The existing tests mock the Inspector to return columns regardless of catalog — that will NOT
catch CR-02. The Wave 0 tests must use a **catalog-discriminating fake**:
- A fake `connection.execute` (or fake cursor) that returns rows ONLY when the SQL contains the
  expected 3-part `` `catalog`.`schema`.`table` `` (or its TSQL pre-transpile form), and returns
  empty otherwise. Assert candidates are non-empty → proves the statement actually carried the
  catalog.
- Assert the executed SQL string never contains `USE CATALOG` (Pitfall 1 guard).
- Assert sqlglot transpile of the 3-part `_qualified_table` equals the expected backtick form
  (no DB needed; pure `sqlglot.transpile`).
