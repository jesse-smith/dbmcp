# Phase 15.1: Code Review Report

**Reviewed:** 2026-05-29
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

Reviewed the cross-catalog metadata threading work that adds a resolved Databricks
catalog to PK discovery, FK candidate search, and column statistics, plus the shared
`CatalogAwareReflector` helper. The structural shape is sound: catalog gating happens
upstream in `resolve_identifier`, the cross-catalog branch is correctly fenced behind
`bool(catalog) and dialect.name == "databricks"`, and the default (non-catalog) paths
are preserved unchanged. WHERE-clause values are consistently parameter-bound.

The central concern is the injection-safety claim that recurs throughout the new
docstrings. The code asserts that "every identifier segment is quoted via
`dialect.quote_identifier` ... so untrusted catalog/schema/table values cannot break
out of the backtick-quoted three-part name (T-15.1-01)." That claim is **false for
Databricks**: `DatabricksDialect.quote_identifier` wraps in backticks but does NOT
escape embedded backticks (contrast `MssqlDialect.quote_identifier`, which escapes
`]` -> `]]`). A catalog/schema/table/column value containing a backtick breaks out of
the quoted identifier. This is a pre-existing gap in `quote_identifier` (the same
weakness exists in `MetadataService`), but Phase 15.1 widens the attack surface by
routing newly user-controllable values (`catalog`, `column_name`) through it on raw
SQL paths while explicitly documenting it as the injection defense.

The remaining findings are correctness/robustness issues in the cross-catalog branches.

## Critical Issues

### CR-01: Databricks `quote_identifier` does not escape backticks — identifier injection on catalog-scoped raw SQL

**File:** `src/db/dialects/databricks.py:124-126`, exercised by `src/analysis/_sql.py:81-83,114-116`, `src/analysis/pk_discovery.py:251-252`, `src/analysis/fk_candidates.py:438-439`, `src/analysis/column_stats.py:412-423`

**Issue:** The phase's documented injection defense is `dialect.quote_identifier`. The Databricks implementation is:

```python
def quote_identifier(self, identifier: str) -> str:
    """Quote using Databricks backticks."""
    return f"`{identifier}`"
```

It does not double or escape an embedded backtick. Every catalog-scoped raw-SQL path in this phase interpolates the result directly:

- `_sql.py` `reflect_columns`: `f"DESCRIBE TABLE {qi(catalog)}.{qi(schema)}.{qi(table)}"`
- `_sql.py` `list_tables`: `f"SHOW TABLES IN {qi(catalog)}.{qi(schema)}"`
- `pk_discovery.py` / `fk_candidates.py`: `f"{qi(self._catalog)}.information_schema"` interpolated into `table_constraints` / `key_column_usage` queries
- `column_stats.py` `_try_describe_extended_stats`: `f"DESCRIBE EXTENDED {qi(catalog)}.{qi(schema)}.{qi(table)} {qi(column_name)}"`

A value such as catalog = `` x`.`information_schema`.`secret` ``  or column = `` c` AS x, (SELECT ...) AS y -- `` survives `quote_identifier` intact and alters the statement. Since `column_name` flows from the unfiltered `columns` tool parameter (`get_column_info`) and `catalog` flows from the `catalog` tool parameter through `resolve_identifier` (which the module header explicitly states "is NOT a SQL sanitizer"), these are attacker-influenced inputs reaching a backtick-quoting routine that cannot contain them. The docstrings asserting safety (`_sql.py:44-50`, `pk_discovery.py:234-238`, `fk_candidates.py:434-436`, `column_stats.py:413-416`) are therefore incorrect.

This is a pre-existing weakness in `quote_identifier` (also relied on by `MetadataService`), but it is load-bearing for this phase and the new paths newly expose `catalog` and `column_name` through it, so it must be fixed before the cross-catalog raw-SQL paths ship.

**Fix:** Make Databricks identifier quoting escape backticks, matching the MSSQL pattern:

```python
def quote_identifier(self, identifier: str) -> str:
    """Quote using Databricks backticks (doubling embedded backticks)."""
    return f"`{identifier.replace('`', '``')}`"
```

Then correct the docstrings to state the defense is "backtick-quoted with embedded backticks doubled." Add a regression test that passes a backtick-bearing catalog/column through `reflect_columns`, `_get_constraint_candidates_cross_catalog`, and `_try_describe_extended_stats` and asserts the emitted SQL keeps the value inside one quoted token.

## Warnings

### WR-01: `_try_describe_extended_stats` swallows all exceptions, masking real failures

**File:** `src/analysis/column_stats.py:425-429`

**Issue:** The bare `except Exception: return None` treats any error from `DESCRIBE EXTENDED` (auth failure, transient network error, an injection-induced syntax error, a genuinely broken connection) identically to "stats not available," then silently falls through to the Tier-2 aggregate path. This hides operational failures and, combined with CR-01, would also mask an injection attempt's syntax error. It also broadens the documented contract ("Returns dict ... None if not [available]") to "None on any failure whatsoever."

**Fix:** Narrow the catch to the expected "feature unavailable" signal. Catch `SQLAlchemyError` only (the rest of the module already imports/uses it at the tool boundary), and prefer inspecting the error to distinguish "DESCRIBE EXTENDED unsupported / no stats" from infrastructure errors that should propagate:

```python
from sqlalchemy.exc import SQLAlchemyError
...
try:
    result = self.connection.execute(text(sql))
    rows = result.fetchall()
except SQLAlchemyError:
    return None
```

### WR-02: `get_basic_stats` / `get_numeric_stats` etc. dereference `row[1]`/`row[2]` after only guarding `row` truthiness

**File:** `src/analysis/column_stats.py:265-269` (and same shape at 293-308, 375-380)

**Issue:** `row = result.fetchone()` then `row[0] if row else 0` / `row[1] if row else 0`. For an aggregate query this is normally safe, but the guard pattern is inconsistent and brittle: if `fetchone()` returns a row with fewer columns than expected (e.g. a transpile that drops a projected column, or a dialect that returns differently shaped output), `row[1]`/`row[2]` raise `IndexError` rather than producing the intended fallback. The fallback `else 0` only protects against `None`, not against a short row. This is low-likelihood on the happy path but is exactly the kind of silent-shape assumption the cross-catalog transpile work makes riskier.

**Fix:** Either trust the aggregate contract and drop the misleading `if row else` guards, or guard width explicitly:

```python
total_rows = row[0] if row and len(row) > 0 else 0
distinct_count = row[1] if row and len(row) > 1 else 0
null_count = row[2] if row and len(row) > 2 else 0
```

### WR-03: Cross-catalog FK "all columns" path reports `is_nullable=True` while structural PK path reports `is_non_null=True` for the same columns

**File:** `src/analysis/fk_candidates.py:266-281` vs `src/analysis/pk_discovery.py:370-378,338-360`

**Issue:** `DESCRIBE TABLE` does not expose nullability, so the two cross-catalog branches make opposite assumptions to compensate:

- FK `get_candidate_columns(pk_candidates_only=False)` sets `is_nullable=True` ("treat as nullable so no candidate is falsely excluded").
- PK `_list_all_columns` cross-catalog sets `is_nullable=False`, and `get_structural_candidates` then emits `is_non_null=True` for any column that passes the uniqueness probe.

The FKCandidateData emitted for a structurally-unique cross-catalog column therefore can carry `target_is_nullable=True` (FK path) while the same column surfaced through `find_pk_candidates` reports `is_non_null=True`. These tools are documented as returning "raw structural metadata," but here both values are fabricated defaults, not reflected facts, and they disagree. A consumer correlating the two tools gets contradictory nullability for one column.

**Fix:** Make the assumption consistent and explicit. Either reflect nullability from the catalog `information_schema.columns` on the cross-catalog branch (it is available there, unlike `DESCRIBE TABLE`), or document a single shared convention (e.g. "nullability is unknown on cross-catalog Databricks; reported as `None`") and surface `None` rather than a fabricated boolean in both `target_is_nullable` and `is_non_null`.

### WR-04: `compute_overlap` interpolates raw column/table names into transpiled SQL with no escaping on the bracket path

**STATUS: RESOLVED (commit b49d525).** Confirmed exploitable, not benign: a `]`-bearing `catalog` (e.g. `"x] WHERE 1=1 --"`) broke out of the bracket during sqlglot's TSQL parse, injecting a live `WHERE` clause and commenting out the rest. Fixed by adding a shared `quote_tsql_identifier()` helper in `_sql.py` (doubles `]` → `]]`, mirroring `MssqlDialect.quote_identifier`) and routing every catalog/schema/table/column bracket interpolation in all three analysis files through it. Red-then-green TDD with a transpile-level containment test (`TestQuoteTsqlIdentifier`); end-to-end check confirms a `UNION SELECT ... --` catalog payload becomes a single inert quoted identifier. Full sweep also verified `src/db/query.py:310` is a non-executed error-message label (false positive) and `query.py` execution paths already route through the now-safe `quote_identifier`.

**File:** `src/analysis/fk_candidates.py:517-552`, and `pk_discovery.py:411-418`, `column_stats.py:255-260,282-289,323-339,365-391`

**Issue:** These build TSQL via f-strings using `[{self.source_column}]`, `[{target_column}]`, and `[{catalog}].[{schema}].[{table}]` without escaping `]`. Unlike `MssqlDialect.quote_identifier` (which doubles `]`), this hand-built bracket quoting does not. On the default MSSQL path a `]`-bearing column name breaks out; on the cross-catalog path the catalog segment is now also hand-bracketed (`f"[{self._catalog}]..."`, fk_candidates.py:519). After `transpile_query`, sqlglot re-quotes to backticks, but a `]` injected before transpile can still alter the parse. This mirrors the pre-existing MSSQL data-query pattern, but Phase 15.1 newly routes the user-controllable `catalog` value through the bracket path in `compute_overlap`, `column_stats._qualified_table`, and `pk_discovery._qualified_table`.

**Fix:** Escape `]` consistently when hand-building bracket identifiers (`name.replace("]", "]]")`), or route these through `dialect.quote_identifier` before transpilation rather than hand-rolling brackets. At minimum escape the newly-threaded `catalog` segment in all three `_qualified_table` / overlap builders.

### WR-05: `get_columns_info` fast-path probe assumes column 0 represents all columns and re-runs DESCRIBE EXTENDED per column

**File:** `src/analysis/column_stats.py:600-623`

**Issue:** The bulk strategy is decided by probing only `columns_to_analyze[0]`: if that single column has precomputed stats, `use_fast_path=True` for every column. Inside the loop, when a later column lacks `DESCRIBE EXTENDED` stats (`desc_stats is None`), the code falls through to `get_column_statistics` (correct), but a column whose `type_info` is not a `TypeEngine` (the string-fallback path, possible on the cross-catalog reflector where types are strings) silently skips the fast path and also falls through — so the fast-path/slow-path decision is effectively per-column anyway, making the upfront probe both redundant and potentially misleading. More importantly, on the cross-catalog Databricks branch `get_column_data_type` returns a string (column_stats.py:194-198), so `isinstance(type_info, sa_types.TypeEngine)` is always False and the fast path is dead code there even though `self._dialect.name == "databricks"` is True — every cross-catalog column silently takes the Tier-2 aggregate path. That is functionally acceptable but means the documented "Databricks fast path" never fires cross-catalog, contradicting the module/tool docstrings.

**Fix:** Either gate the probe on `not self._is_cross_catalog_databricks` and document that the DESCRIBE EXTENDED fast path is default-catalog only, or make the cross-catalog reflector return a `TypeEngine` so the fast path can engage. Drop the single-column probe in favor of the already-per-column fall-through, or document why column 0 is representative.

## Info

### IN-01: `list_tables` row-shape fallback is inconsistent with the documented contract

**File:** `src/analysis/_sql.py:119`

**Issue:** `return [row[1] if len(row) > 1 else row[0] for row in rows]`. The docstring states SHOW TABLES returns `(database, tableName, isTemporary)` and `row[1]` is the table name. The `len(row) > 1` fallback to `row[0]` is defensive padding for a shape the docstring says never occurs; if it ever did occur, returning the database name as a table name would produce silently wrong results rather than failing. Prefer asserting the expected shape or documenting why the fallback exists.

### IN-02: Lazy `from src.db.metadata import MetadataService` duplicated in three tool functions

**File:** `src/mcp_server/analysis_tools.py:126-128,275-277,442-444`

**Issue:** The same in-function import appears three times. If it is lazy to avoid a circular import, that rationale belongs in a comment; if not, it should be a module-level import. As-is it is unexplained duplication.

**Fix:** Hoist to module top-level if no cycle exists, or add a one-line comment noting the cycle being broken.

### IN-03: Identical cross-catalog branch scaffolding duplicated across three tools

**File:** `src/mcp_server/analysis_tools.py:112-147,262-295,429-462`

**Issue:** The `cross_catalog = bool(resolved.catalog) and dialect is not None and dialect.name == "databricks"` computation plus the `if cross_catalog: MetadataService.table_exists(...) else: inspector.get_table_names/get_view_names` existence block is copy-pasted in all three tools with only the error-message wording differing slightly (line 146 says "in schema 'X'", line 135 omits it). The same `_cross_catalog` predicate is also independently recomputed inside `PKDiscovery`, `FKCandidateSearch`, and `ColumnStatsCollector`. This is the Rule-of-Three threshold crossed; consider a shared `resolve_and_check_table_exists(...)` helper to keep the three tools and the predicate in sync.

**Fix:** Extract the existence-check block (and ideally the `cross_catalog` predicate) into a shared helper used by all three tools.

### IN-04: Inconsistent "table not found" error messages between cross-catalog and default paths

**File:** `src/mcp_server/analysis_tools.py:135 vs 146` (and the analogous pairs at 284/294, 451/461)

**Issue:** The cross-catalog branch emits `Table 'schema.table' not found` while the default branch emits `Table 'table' not found in schema 'schema'`. Same failure, two formats. Downstream consumers parsing error_message see drift depending on which path executed.

**Fix:** Use one message template for both branches.
