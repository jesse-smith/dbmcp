# Phase 15.1 Plan 05: Thread Resolved Catalog Through Analysis Tools Summary

Wired `resolved.catalog` end-to-end through all three analysis tools (`find_pk_candidates`, `find_fk_candidates`, `get_column_info`) at the MCP boundary, so the explicit `catalog` arg now reaches the Wave 1-3 catalog-aware analysis classes and the catalog-aware existence check (closes CR-02/IDENT-08 at the integration seam).

## What Was Built

### Task 1 (RED) — Boundary tests
Added a `TestCrossCatalogWiring` class to `tests/unit/test_async_tools.py` with 7 tests asserting:
- `catalog` threads into `PKDiscovery` / `FKCandidateSearch` / `ColumnStatsCollector` on a Databricks connection with `catalog="cerner_src"`.
- `find_pk_candidates` / `find_fk_candidates` use the catalog-aware `MetadataService.table_exists` existence check on the cross-catalog branch (not the catalog-blind Inspector path).
- `find_fk_candidates` reads the source-column type via `CatalogAwareReflector.reflect_columns` (not `inspector.get_columns`) on the cross-catalog branch.
- Default path (`catalog=None`): all three construct their class with `catalog=None` and use the Inspector existence path (no `MetadataService.table_exists`).
- MSSQL `catalog` rejection (resolver gate) unchanged for all three tools.

6 of 7 tests failed as expected (RED); the MSSQL-rejection test passed immediately because the resolver gate was already correct and untouched.

### Task 2 (GREEN) — Wiring
In each of the three `_sync_work` functions, computed `cross_catalog = bool(resolved.catalog) and dialect is not None and dialect.name == "databricks"` once after `resolve_identifier`, then:
- **get_column_info**: passed `catalog=resolved.catalog if cross_catalog else None` into `ColumnStatsCollector`. Existence check was already catalog-aware; replaced its inline predicate with the shared `cross_catalog` flag.
- **find_pk_candidates**: added the catalog-aware `MetadataService.table_exists` existence branch (mirroring `get_column_info`), falling back to the Inspector path on the default branch; passed `catalog=...` into `PKDiscovery`.
- **find_fk_candidates**: same existence-check wrap; on the cross-catalog branch the source-column type is read via `CatalogAwareReflector(connection, dialect).reflect_columns(catalog, schema, table)` (returns `data_type` as a string) instead of `inspector.get_columns`; passed `catalog=...` into `FKCandidateSearch`.
- Imported `CatalogAwareReflector` into `analysis_tools.py`.
- Updated the stale docstring notes in all three tools (they previously claimed the Inspector/stats bind to the connection's default catalog and required connecting with that catalog as default — now they thread end-to-end).

Updated two existing 3-part-name tests (`test_find_pk_candidates_three_part_routes_resolved`, `test_find_fk_candidates_three_part_routes_resolved`) that asserted the old Inspector-only behavior for a cross-catalog name — they now assert the new `MetadataService.table_exists` + `CatalogAwareReflector` routing, matching the contract change in the plan's must-haves.
