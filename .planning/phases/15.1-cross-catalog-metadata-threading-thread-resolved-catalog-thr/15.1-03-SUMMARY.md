# Phase 15.1 Plan 03: Catalog-aware FKCandidateSearch Summary

Threaded `resolved.catalog` into `FKCandidateSearch` so `find_fk_candidates` on a non-default
Databricks catalog reflects the source column from that catalog AND searches FK targets within
that catalog only — eliminating the CR-02 silent mis-targeting for FK discovery (IDENT-08).
FK search is multi-table, so threading covers both the source-side overlap reads and the
target-table universe (RESEARCH Pitfall 3).

## What Was Built

`FKCandidateSearch.__init__` now accepts `catalog: str | None = None`, storing `self._catalog`
and a `_cross_catalog` guard (`bool(catalog) and dialect.name == "databricks"`). When set, the
cross-catalog branch covers every read that previously assumed the connection default catalog:

- **Target-table universe** (`get_target_tables` → new `_get_target_tables_cross_catalog`):
  enumerated via `CatalogAwareReflector.list_tables(catalog, schema)` = `SHOW TABLES IN
  catalog.schema`, scoped to the resolved catalog ONLY (Pitfall 3, Open Q1 KISS). Explicit-list
  and LIKE-pattern client filtering and source-table exclusion mirror the Inspector path.
- **Nested PKDiscovery** (`get_candidate_columns`, `pk_candidates_only=True`): constructed with
  `catalog=self._catalog`, so target PK/UNIQUE candidacy is discovered from the resolved catalog
  (delegates to the Plan 02 cross-catalog PKDiscovery path).
- **All-columns reflection** (`pk_candidates_only=False`): routed to
  `CatalogAwareReflector.reflect_columns(catalog, schema, table)` (DESCRIBE TABLE) over the
  class's existing live connection. DESCRIBE exposes no nullability, so columns are treated as
  nullable (no FK candidate falsely excluded).
- **Constraint metadata** (`get_column_metadata` → new `_get_constraints_cross_catalog`):
  PK/UNIQUE read from the catalog's `information_schema.table_constraints` JOIN
  `key_column_usage`, catalog segment backtick-quoted via `dialect.quote_identifier` (identifiers
  cannot be parameter-bound; schema/table/column are bound). Index check is unchanged —
  Databricks `supports_indexes=False` already leaves `has_index=None`.
- **Value overlap** (`compute_overlap`): source AND target table references become 3-part
  TSQL-bracket names `[catalog].[schema].[table]` when a catalog is set, which
  `transpile_query(read="tsql")` renders as `` `cat`.`sch`.`tbl` `` for Databricks. Brackets, not
  backticks, are used because the string is fed through the TSQL→dialect transpiler (Pitfall 4).

The default-catalog path (`catalog=None`) is unchanged: Inspector / INFORMATION_SCHEMA reads and
2-part references are preserved verbatim. No catalog-switching statement is emitted on any path
(stateless 3-part qualification; T-15.1-07).
