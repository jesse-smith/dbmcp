# Phase 13.1: Close v2.0 gap — thread dialect through schema_tools/query_tools entry points — Context

**Gathered:** 2026-05-06
**Status:** Ready for planning
**Source:** `/gsd-discuss-phase` (grounded in `.planning/v2.0-MILESTONE-AUDIT.md`)

<domain>
## Phase Boundary

Close the 3 integration wiring gaps identified in the v2.0 milestone audit so the stored dialect is threaded end-to-end through the MCP tool entry points:

- **WIRING-01** (medium) — `src/mcp_server/schema_tools.py:170` — `connect_database` constructs one-shot `MetadataService(engine)` without the resolved dialect; only the `schema_count` in the connect response is affected (subsequent tool calls use the correctly-dialected `_get_metadata_service` helper). Affects REQ **META-01**.
- **WIRING-02** (high) — `src/mcp_server/query_tools.py:91` — `get_sample_data` constructs `MetadataService(engine)` and `QueryService(engine, metadata_service=metadata_svc)` without dialect. For Databricks/generic connections `QueryService` auto-infer yields `dialect="tsql"` + empty `safe_procedures` → silent validation degradation in `validate_query`. Affects REQs **VALID-01, VALID-02, META-05**.
- **WIRING-03** (low/cosmetic) — `src/db/connection.py ConnectionManager.connect()` MSSQL path — `connect_with_config`'s MSSQL branch calls `self.connect()` which re-creates `MssqlDialect()` internally, ignoring any caller-supplied dialect. No behavioral impact today; latent coupling.

**Also resolves:** VALID-01 / VALID-02 integration-boundary degradation on `get_sample_data` path (audit-flagged as "partial"). After this phase, both flow rows in the milestone audit move from PARTIAL → COMPLETE.

**Out of scope:** v2.0 tech debt items tracked in `.planning/v2.0-MILESTONE-AUDIT.md` (ROADMAP text cleanup, REQUIREMENTS TEST-03 wording, SUMMARY.md frontmatter inconsistencies, `src/metrics.py` ruff warning). Those remain tech debt.

</domain>

<decisions>
## Implementation Decisions

### WIRING-03 scope
- **Fix WIRING-03 now.** Thread caller dialect through `ConnectionManager.connect()` for consistency — removes latent coupling even though no behavioral impact today. Phase title already scopes it in.
- Fix style: minimal — update signature to accept `dialect` and use it instead of re-instantiating `MssqlDialect()`. Preserve existing default behavior for callers that don't pass dialect.

### get_sample_data fix style (WIRING-02)
- **Reuse `_get_metadata_service(connection_id)`** helper in `src/mcp_server/schema_tools.py:69-74`. Single source of truth for dialect resolution. DRY with existing correctly-dialected path.
- Construct `QueryService(engine, dialect=dialect, metadata_service=metadata_svc)` using the same dialect the helper already fetches. Expose or re-derive dialect in the call site so both services get it.
- If the existing helper is private-to-schema_tools, promote it (or the dialect-fetch step) so `query_tools.get_sample_data` can call it without a cross-module private reach. Planner decides: promote helper to a shared `_services.py`-style module, or duplicate the one-line `conn_manager.get_dialect(connection_id)` call at the query_tools site. Either is acceptable — optimize for readability.

### connect_database fix style (WIRING-01)
- Pass `dialect=dialect` (already in local scope for both TOML and URL branches at schema_tools.py:152 / :162) into `MetadataService(engine, dialect=dialect)` at line 170. One-line fix.

### Regression test strategy
- **Assert-dialect unit tests per call site.** For each of the 3 fix sites, add a test that mocks `conn_manager.get_dialect` (or checks the dialect value in scope) and asserts the service was constructed with the correct dialect. Fast, deterministic, catches regression even if higher-level behavior tests miss due to dialect fixture gaps.
- Minimum 3 new tests (one per wiring site). Planner may add more if helper promotion changes the unit boundary.

### WIRING-01 test scope
- **Include schema_count behavior test.** Add a test asserting `connect_database` response `schema_count` is populated correctly for a generic URL connection (where `engine.dialect.name` does not match the `"generic"` registry key). This is the concrete user-visible consequence of WIRING-01; asserting the service construction alone doesn't prove the response surfaces the right count.

### Claude's Discretion
- File-level organization if helper promotion is chosen (which module, naming).
- Whether to add a dialect-threading regression test at the `validate_query`-from-`get_sample_data` seam (behavior-level), in addition to the unit assertion — planner may add if it's trivial with existing fixtures.
- Whether to include a test for WIRING-03 beyond the construction assert (e.g., verify `connect_with_config` MSSQL path doesn't drop a caller-supplied dialect). Audit says no behavioral impact; planner picks.
- Exact test file placement (existing `tests/mcp_server/test_schema_tools.py`, `tests/mcp_server/test_query_tools.py`, `tests/db/test_connection.py` — or new file).

</decisions>
