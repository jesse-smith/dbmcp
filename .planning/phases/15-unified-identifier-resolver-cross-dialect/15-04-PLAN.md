<objective>
Wire the resolver (Plan 03) into the three schema tools — `list_schemas`, `list_tables`, `get_table_schema` — at their `@mcp.tool` boundaries (D-03), and finish the SC4 dbo sweep on their signatures. These three tools ALREADY have a `catalog` param and a `ValueError -> error_message` boundary; this plan adds the resolver call inside each `_sync_work`, removes the `schema_name="dbo"` defaults, and flips the `catalog` docstrings from "Ignored for non-Databricks" to "rejected" (D-07, backward-incompatible).

`list_schemas` has NO `table_name` (it lists schemas for a catalog), so it cannot feed `resolve_identifier`. It enforces the D-07 catalog gate by importing and calling the SHARED `_assert_catalog_allowed(catalog, dialect)` exported by `src/db/identifiers.py` (Plan 03, Blocker-3 resolution). This plan IMPORTS `_assert_catalog_allowed` from `src/db/identifiers.py` but does NOT edit it (Plan 03 authors the helper); the dependency is captured by `depends_on: [15-03]`, so identifiers.py is intentionally NOT in this plan's `files_modified` (import-only, no write).

Purpose: All three schema tools stop interpreting `table_name`/`schema_name`/`catalog` themselves and instead pass resolver-normalized parts to `MetadataService`, with one shared catalog gate. The catalog gate (D-07) now makes MSSQL/generic `catalog` an error instead of a silent ignore.
Output: 1 modified tool file (3 tools), extended boundary tests.

⚠ Backward-incompatible (RESEARCH Pitfall 3): MSSQL/generic callers that previously passed `catalog` to these 3 tools and got silent-ignore now get a status=error response. This is intentional and unified across all 7 tools. Size and test it explicitly.

NOTE: this plan EDITS only `src/mcp_server/schema_tools.py` and `tests/unit/test_async_tools.py`. `src/db/identifiers.py` is an import-only dependency (helper authored in Plan 03) — do NOT author or alter the gate here.
</objective>
