<objective>
Add the `catalog` parameter to `get_sample_data` (IDENT-05) and `get_column_info` (IDENT-06) — the two tools that currently lack it — route both through the resolver at their `@mcp.tool` boundaries (D-03), drop their `schema_name="dbo"` signature defaults, and thread the resolved catalog into the SQL/inspector paths so a Databricks 3-part `table_name` works end-to-end without `USE CATALOG` (SC3). The catalog gate (D-07) is enforced by the resolver: MSSQL/generic callers passing `catalog` get a status=error response.

Purpose: Unify these two tools with the three schema tools so all five share one resolver and one catalog gate. Close the IDENT-05 SC3 gap where `bmtct.ml_infections_ref.mv_fever_episodes` previously needed catalog workarounds.
Output: 2 modified tool files + query.py catalog threading, extended boundary tests.

SC3 sizing note: `query.py:get_sample_data` currently builds `full_table_name` as `schema.table` (line 119) with NO catalog — for a Databricks 3-part identifier the catalog must reach the SQL. `get_column_info` uses the SQLAlchemy inspector, which binds to the engine's default catalog (set at connect time, Phase 14 IDENT-01); for cross-catalog access the resolved catalog must reach the inspector path, mirroring the existing `metadata.py` Databricks catalog gate (`table_exists`/`get_table_schema`).
