<objective>
Build the cross-dialect identifier resolver: a new standalone module `src/db/identifiers.py` exporting a frozen `ResolvedIdentifier(catalog, schema, table)` dataclass (D-02) and a top-level `resolve_identifier(table_name, schema_name, catalog, dialect)` function (D-01). This is the heart of the phase (IDENT-03 depth parsing, IDENT-04 conflict detection, D-07 catalog gate, IDENT-07 default-schema fill). It is pure business logic with defined I/O — a TDD candidate. Tests are written and made to fail BEFORE the implementation.

This module ALSO exports a small reusable catalog-gate — `_assert_catalog_allowed(catalog, dialect)` plus a module-level `CATALOG_GATE_MESSAGE` — because three call sites need the bare catalog gate WITHOUT a `table_name`: the resolver itself, `list_schemas` (no table_name), and `find_pk_candidates`/`find_fk_candidates` (D-14, now namespace-aware). Rule of Three → one shared helper is the DRY-correct call. `resolve_identifier` calls `_assert_catalog_allowed` internally; the table_name-less tools import and call it directly.

Purpose: One dialect-agnostic layer parses + validates + normalizes identifiers so all seven tools (Plans 04/05/06) stop interpreting `table_name`/`schema_name`/`catalog` themselves. Dialect facts come from the properties added in Plan 01.
Output: 1 new source module, 1 new exhaustive parametrized test file.

Critical correctness trap (RESEARCH.md Pitfall 1): depth MUST be measured by `len(Table.parts)`, NOT by `Table.name/db/catalog`. `to_table("a.b.c.d")` does NOT raise — it folds the 4th segment into a Dot node and the attributes report only 3 slots, silently swallowing over-depth. Counting `.parts` is the only correct depth check.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| tool caller → resolve_identifier (table_name/schema_name/catalog) | Untrusted user-supplied identifier strings cross here. The resolver SPLITS and validates structure; it is NOT a SQL sanitizer. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-04 | Tampering | SQL injection via crafted `table_name` (e.g. `a; DROP TABLE b`) | mitigate (downstream) | sqlglot.to_table parses `a; DROP TABLE b` as ONE literal identifier — no split, no error (RESEARCH §Security). The resolver does NOT sanitize; downstream `dialect.quote_identifier` (query.py:119, metadata.py) remains the injection defense and is untouched. The resolver MUST NOT bypass quote_identifier — it only returns decomposed string parts that flow into the existing quoting path. |
| T-15-05 | Denial of Service | Malformed input (`""`, `"a."`) raising uncaught sqlglot.ParseError | mitigate | Catch `sqlglot.ParseError` and re-raise as `ValueError` (A1) so it maps cleanly through the existing `except ValueError -> error_message` tool boundary rather than surfacing as an unexpected exception. |
| T-15-06 | Information Disclosure | Conflict/depth error messages echo user identifier segments | accept | Messages name the conflicting segment (D-05 templates) for actionable errors; segments are caller-supplied identifiers, not secrets. Low risk. |
| T-15-SC | Tampering | pip installs | accept | No package installs; `sqlglot` 30.7.0 already pinned (RESEARCH Package Legitimacy Audit: not applicable). |
