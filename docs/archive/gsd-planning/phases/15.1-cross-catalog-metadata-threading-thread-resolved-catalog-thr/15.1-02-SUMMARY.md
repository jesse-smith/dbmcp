# Phase 15.1 Plan 02: Catalog-aware PKDiscovery Summary

Threaded `resolved.catalog` into `PKDiscovery` so `find_pk_candidates` on a non-default
Databricks catalog reads that catalog's columns/constraints instead of silently
mis-targeting the connection default (CR-02 / IDENT-08).

## What Was Built

`PKDiscovery.__init__` now accepts `catalog: str | None = None`. When set:

- `_qualified_table` becomes the 3-part TSQL-bracket form `[catalog].[schema].[table]`,
  which `transpile_query(read="tsql")` renders as `` `cat`.`sch`.`tbl` `` for Databricks.
  Brackets (not backticks) are used because the string is fed through the TSQL→dialect
  transpiler (Pitfall 4 — pre-quoting with backticks would double-quote/mis-transpile).
- A `_cross_catalog` guard (`bool(catalog) and dialect.name == "databricks"`) routes
  constraint and column reads to a new cross-catalog path:
  - Columns: `CatalogAwareReflector(self.connection, self._dialect).reflect_columns(...)`
    — DESCRIBE TABLE over the analysis class's existing live connection (not a fresh
    `engine.connect()`), returning `{"name","data_type"}` dicts.
  - PK/UNIQUE constraints: catalog-scoped `information_schema.table_constraints` JOIN
    `key_column_usage`, with `catalog.information_schema` and every identifier segment
    backtick-quoted via `dialect.quote_identifier`.
- The structural uniqueness path (`_column_is_unique`) is fixed automatically — it already
  interpolates `_qualified_table` and runs through `transpile_query`.

The default-catalog path (`catalog=None`) is unchanged: Inspector reads and the 2-part
`_qualified_table` are preserved verbatim. No catalog-switching statement is emitted on any
path (stateless 3-part qualification; T-15.1-04).
