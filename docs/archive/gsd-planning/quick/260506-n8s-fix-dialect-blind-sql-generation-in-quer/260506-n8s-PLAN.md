<objective>
Replace `QueryService._build_*_query` dialect-blind branching (`self._dialect is None` ≈ SQLite, else SQL Server) with dialect-provided builders. Databricks currently receives `SELECT TOP (...)` and fails with PARSE_SYNTAX_ERROR. After this fix:

- MSSQL emits `SELECT TOP (n) ...` / `TABLESAMPLE (n ROWS)` / ROW_NUMBER+TOP modulo — unchanged.
- Databricks and Generic emit `SELECT ... LIMIT n`, `ORDER BY RAND() LIMIT n` for statistical sampling (no TABLESAMPLE), and ROW_NUMBER+LIMIT modulo.
- `QueryService` delegates via `self._dialect.build_sample_query(...)` — no more `is None` branching in the three sampling builders.
- SQLite path for tests: `GenericDialect` with `sqlite` sqlglot name already flows through as Generic, so the `_dialect is None` fallback in `_build_*` methods is removed. Tests that rely on `dialect=None` semantics are updated (or the `None` path retained as a dumb pass-through to Generic semantics).

The three private `_build_*_query` methods collapse into one-liners that call `self._dialect.build_sample_query(method, full_table_name, column_sql, sample_size)`, with a single `None`-dialect fallback (for legacy tests) that delegates to a default-Generic-like implementation.

Regression gate: Databricks dialect-level unit tests verify SQL strings contain `LIMIT n` and never contain `TOP (` or `TABLESAMPLE`. MSSQL dialect tests verify `TOP (n)` is still emitted. Generic dialect test verifies `LIMIT n` for all three sampling methods.

Also update `.planning/phases/13.1-.../deferred-items.md` marking this item resolved with the commit SHA.
</objective>

   - TOP: `SELECT TOP ({n}) {cols} FROM {table}`
   - TABLESAMPLE: `SELECT TOP ({n}) {cols} FROM {table} TABLESAMPLE ({n} ROWS)`
   - MODULO: the `ROW_NUMBER() OVER (ORDER BY (SELECT NULL)) ... WHERE _rn % CASE WHEN ...` SQL currently in `_build_modulo_query` MSSQL branch.
   - Raise `ValueError(f"Unknown sampling method: {method}")` for unknown methods.

3. **`src/db/dialects/databricks.py`** — implement on `DatabricksDialect`:
   - TOP: `SELECT {cols} FROM {table} LIMIT {n}`
   - TABLESAMPLE: `SELECT {cols} FROM {table} ORDER BY RAND() LIMIT {n}` (Databricks supports `TABLESAMPLE` but not with `ROWS` semantics reliably — `ORDER BY RAND()` is portable and satisfies "statistical" intent).
   - MODULO: `SELECT {cols} FROM (SELECT *, ROW_NUMBER() OVER (ORDER BY 1) AS _rn, COUNT(*) OVER () AS _total FROM {table}) _sampled WHERE _rn % CASE WHEN _total / {n} < 1 THEN 1 ELSE _total / {n} END = 0 LIMIT {n}`

4. **`src/db/dialects/generic.py`** — implement on `GenericDialect` identical to Databricks except TABLESAMPLE uses `ORDER BY RANDOM()` (SQLite/Postgres compatible) instead of `RAND()`, and MODULO uses `ORDER BY ROWID` for the window-spec when `self._sqlglot_dialect == "sqlite"` else `ORDER BY 1`. Keep it simple: one branch on `sqlglot_dialect == "sqlite"`.

5. **`src/db/query.py`** — collapse `_build_top_query`, `_build_tablesample_query`, `_build_modulo_query`:
   - Each becomes a thin wrapper: `return self._dialect.build_sample_query(SamplingMethod.X, full_table_name, column_sql, sample_size)`.
   - Preserve `self._dialect is None` fallback: when no dialect is available, instantiate `GenericDialect()` lazily (or use a module-level default) and delegate. This preserves behavior for tests constructing `QueryService(engine)` without a dialect while keeping a single source of truth.
