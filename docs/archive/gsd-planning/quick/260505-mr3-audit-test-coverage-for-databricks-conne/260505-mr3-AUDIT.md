# Databricks connect_with_config Test Coverage Audit

**Date:** 2026-05-05
**Source todo:** `.planning/todos/pending/2026-04-28-audit-databricks-test-coverage.md`
**Context:** Phase 11 shipped with 607 tests; `DatabricksDialect.create_engine` signature drift still slipped through and was fixed reactively in commit `4fdaa90` (quick-260428-jr7).

## Summary

**Verdict: NOT A BLOCKER for further Databricks work — post-fix coverage is adequate.**

The Phase 11 test suite had five Databricks `connect_with_config` tests, but every one mocked `ConnectionManager.connect_with_url` (the layer *above* the dialect), so the URL-vs-kwargs contract between `connect_with_url` and `DatabricksDialect.create_engine` was never exercised. Bug B (signature mismatch) and Bug A (missing env-var resolution on `host`/`http_path`) were both structurally invisible. The fix commit `4fdaa90` added `tests/unit/test_connect_with_config_databricks.py` (2 tests that exercise the real dialect dispatch path) and added 2 env-var regression tests in `test_connect_tool.py` — both gaps are now covered. One residual observation: the plan's `<interfaces>` block describes a `connect_with_config → connect_with_url → dialect.create_engine` call path, but post-fix the Databricks branch bypasses `connect_with_url` entirely and calls `dialect.create_engine` directly (`src/db/connection.py:487-493`). This is worth flagging for plan-writer accuracy but does not change the verdict.

## Audit Questions

### Q1 — Is there an integration-style test covering connect_with_config → dialect.create_engine for Databricks?

**At Phase 11 merge time (commit `551b99d`): NO.** Every Databricks `connect_with_config` test mocked `connect_with_url`, so the real `DatabricksDialect.create_engine` never ran. Smoking-gun citation from the Phase-11 commit:

- `tests/unit/test_connect_tool.py:265` (at `551b99d`) — `test_databricks_config_routes_to_connect_with_url` — asserts only that the URL string passed to the mocked `connect_with_url` contains the expected substrings. The mocked boundary is `patch.object(cm, "connect_with_url")`.

**Post-fix (current HEAD): YES.** Bucket-A tests now exist in `tests/unit/test_connect_with_config_databricks.py`:

- `tests/unit/test_connect_with_config_databricks.py:44` — `test_connect_with_config_resolves_env_vars_and_calls_dialect_with_kwargs` — patches `DatabricksDialect.create_engine` at the *dialect class* level (not the `ConnectionManager`), so the real `connect_with_config` dispatch executes end-to-end and the kwargs passed to `create_engine` are asserted (see lines 60-66 for the spy shape and 76-84 for assertions).
- `tests/unit/test_connect_with_config_databricks.py:90` — `test_connect_with_config_databricks_signature_matches_dialect` — calls the real `DatabricksDialect.create_engine` with exactly the kwargs the caller passes and asserts it does not raise. Also asserts missing `host` still raises `ValueError` (line 123).

### Q2 — If yes, why did it not catch the signature mismatch?

The pre-fix tests all used **bucket B (mock-heavy)** mocking shape that hid Bug B. The specific drift-hiding patterns in the Phase-11 suite:

1. **`patch.object(cm, "connect_with_url")` — mocks the caller, not the callee.** Example at `tests/unit/test_connect_tool.py:278` (Phase-11 revision): `with patch.object(cm, "connect_with_url") as mock_connect_url: ... mock_connect_url.assert_called_once()`. The test asserts the URL *string* handed to the mock, never that a real dialect could consume it. Because `connect_with_url` was mocked, `dialect.create_engine(sqlalchemy_url=url)` never ran.
2. **MCP-tool-level tests mock the whole `ConnectionManager`.** Example: `tests/unit/test_connect_tool.py:193-212` — uses `mock_cm = MagicMock()` with `mock_cm.connect_with_config.return_value = mock_conn`. Tests the MCP tool's routing wiring, not the connection code path.
3. **Bucket C (unit-only) tests exercised only the kwargs contract in isolation.** `tests/unit/test_databricks_dialect.py:78,119,142,166` — all four `test_create_engine_*` tests pass `host=…, http_path=…` kwargs directly, matching `DatabricksDialect.create_engine`'s actual signature. They confirm the dialect-side contract works but never connect it to what the caller passes.

Net effect: the URL produced by `connect_with_config` and the kwargs expected by `DatabricksDialect.create_engine` lived in separate tests with a mocked seam between them. The drift was a pure integration-contract hole.

### Q3 — Env-var substitution for Databricks host/http_path?

**At Phase 11 merge: NO.** The Phase-11 commit `551b99d` added `test_databricks_config_resolves_env_var_token` (`test_connect_tool.py:294` at that SHA) for token only — matching the bug in `connection.py` where only `token` went through `resolve_env_vars`. The test and the bug agreed; no test would fail.

**Post-fix: YES.** Added in `4fdaa90`:

- `tests/unit/test_connect_tool.py:323` — `test_databricks_config_resolves_env_var_host_and_http_path` — sets `${DBX_HOST}` / `${DBX_PATH}` and asserts resolved kwargs (line 343-344).
- `tests/unit/test_connect_with_config_databricks.py:44` (already cited) — resolves `${DATABRICKS_HOST}`, `${DATABRICKS_HTTP_PATH}`, and `${DATABRICKS_TOKEN}` together.

Comparison to the MSSQL pattern: the closest analog is `tests/unit/test_config.py:497` (`test_connection_with_env_var_password_not_resolved`), which covers a different layer (config parsing does *not* resolve env vars — resolution happens in `connect_with_config`). The Databricks env-var regression tests now follow the correct "resolve at connect time" pattern, matching `connection.py:463-467`.

## Gaps and Proposed Tests

1. **Gap:** No test asserts `catalog` / `schema_name` env-var substitution — the regression tests pass literal strings for these. If `resolve_env_vars` is ever removed from line 466 or 467, no test fails.
   - **Proposed test:** `tests/unit/test_connect_with_config_databricks.py::test_env_var_substitution_for_catalog_and_schema`
   - **What it asserts:** Given `catalog="${DBX_CATALOG}"` and `schema_name="${DBX_SCHEMA}"`, the kwargs captured by the spy have the resolved values.
   - **Effort:** small (≤20 lines, uses the existing `spy_create_engine` and `_make_engine_spy` helpers in the same file).

2. **Gap (plan-level, not code):** The plan's `<interfaces>` block describes `connect_with_config → connect_with_url → dialect.create_engine(sqlalchemy_url=…)` as the live Databricks call path. Post-fix, the Databricks branch in `connection.py:462-513` does **not** route through `connect_with_url` — it calls `dialect.create_engine(**kwargs)` directly at `connection.py:487-493`. Future audits/plans should cite the direct-dispatch pattern.
   - **Proposed test:** none (documentation/plan-hygiene issue).
   - **Effort:** n/a.

3. **Gap:** No test exercises the `ConnectionError` wrapping path at `connection.py:494-502` for Databricks (when the dialect raises `SQLAlchemyError`).
   - **Proposed test:** `tests/unit/test_connect_with_config_databricks.py::test_sqlalchemy_error_wrapped_as_connection_error`
   - **What it asserts:** When `DatabricksDialect.create_engine` raises `SQLAlchemyError`, `connect_with_config` raises `ConnectionError` containing the host.
   - **Effort:** small (≤15 lines).

## Verdict

- **Blocker for further Databricks work?** **NO.**
- **Rationale:** The structural test-coverage hole that allowed Bug A and Bug B to ship through Phase 11 has been closed by the `4fdaa90` fix commit. The two Bucket-A tests in `test_connect_with_config_databricks.py` exercise the real `DatabricksDialect` dispatch path and lock in both the kwargs signature (Bug B guard) and env-var resolution for `host`/`http_path`/`token` (Bug A guard). Residual gaps are minor and non-safety-critical.
- **Recommended follow-up:** Defer the three small gaps above to a follow-up todo — they are nice-to-have regression hardening, not blockers.
